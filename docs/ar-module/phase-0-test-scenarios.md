# Phase 0 — 실기 테스트 시나리오

> 배포된 프로덕션에서 **모바일 실기기**로 `/ar-tech-test` 를 검증.
> 결과는 `ar-test-matrix.md` 표에 단말별 1행씩 기록.
> 치명적 이슈 발견 시 즉시 보고 → Phase 1 착수 전 수정.

---

## 0. 사전 준비

### 기기 준비
- [ ] iOS 16.4 이상 단말 (iPhone 12 이상 권장) — **필수 3대 이상**
- [ ] iOS 16.4 미만 단말 1대 — **가능하면** (Level 4 폴백 검증)
- [ ] Android Chrome 중급 단말 (Pixel 5 이상 또는 Galaxy S20 이상)
- [ ] Android 저가 단말 (Galaxy A 시리즈 등) — **가능하면** (Level 3 트리거 확인)

### 접속 조건
- Vercel 최신 배포(`f8e161c` 이후) 완료 확인
- 실제 프로덕션 도메인으로 접속 (HTTPS 필수)
- **권한 초기화 상태에서 시작** — 이전 세션 권한이 남아있으면 정확한 흐름 검증 불가
  - iOS: 설정 > Safari > 고급 > 웹사이트 데이터 > 도메인 찾아 삭제
  - Android Chrome: 사이트 설정 > 도메인 > 권한 초기화

### 진입 경로 확인
두 가지 진입 경로 모두 정상 도달하는지 확인:
- **A**: 홈 페이지 → 하단 내비게이션 "**탐험**" 탭 → 자동 이동
- **B**: URL 직접 입력 `https://<도메인>/ar-tech-test`

## 1. 테스트 시나리오 (단말별 수행)

각 시나리오 후 기대 결과와 일치하는지 확인. 불일치 시 `이슈` 컬럼에 기록.

### 시나리오 1 — 정상 흐름 (Level 1 진입)

**대상**: iOS 16.4+ / Android Chrome 중급 단말 (모든 권한 허용)

**단계**:
1. 홈 → "탐험" 탭
2. 인트로 화면 확인 (제목/설명/3개 권한 리스트)
3. "테스트 시작" 버튼 탭
4. 위치 권한 다이얼로그 → **허용**
5. 카메라 권한 다이얼로그 → **허용**
6. (iOS만) 자이로 권한 다이얼로그 → **허용**
7. 카메라 피드가 전체화면 배경으로 표시되는지 확인
8. 회전하는 **BoxGeometry 큐브**가 카메라 위에 오버레이되는지 확인
9. 기기를 좌우로 기울여서 큐브 위치가 **시차 이동**되는지 확인
10. 하단 진단 패널에 값이 채워졌는지 확인

**기대 결과**:
- Level 배지: `Level 1` (초록)
- GPS/Camera/Orientation 모두 `granted` (초록 배지)
- WebGL: `WebGL 2` (또는 `WebGL 1`)
- Camera res: `1280×720` 또는 유사 해상도
- FPS: 30 이상 (이상적 55~60)
- Safe Area: iOS 노치 단말은 `T 47px` 정도 표시

**확인 포인트**:
- 좌상단 백 버튼이 안전영역(노치/다이내믹 아일랜드) 침범 없이 표시
- 백 버튼 탭 → 홈 또는 직전 페이지로 정상 복귀 + 카메라 LED 꺼짐

---

### 시나리오 2 — 자이로 거부 → Level 2 폴백

**대상**: iOS (자이로 권한만 거부)

**단계**:
1. 인트로 → "테스트 시작"
2. 위치 허용 → 카메라 허용 → **자이로 거부**
3. 화면 확인

**기대 결과**:
- 상단 중앙 배너 "폴백 진입 — Level 2" 표시
- Level 배지: `Level 2` (노란)
- Orientation 배지: `denied`
- 큐브는 화면 중앙 고정 (기기 기울여도 위치 변화 없음)
- 큐브 자체 회전 애니메이션은 계속 동작

---

### 시나리오 3 — 카메라 거부 → Level 4 대체 경로

**대상**: iOS 또는 Android (카메라 거부)

**단계**:
1. 인트로 → "테스트 시작"
2. 위치 허용 → **카메라 거부**
3. 자이로 다이얼로그는 iOS에서만 나타남 (허용/거부 무관)
4. 화면 확인

**기대 결과**:
- 전체화면 **"AR을 사용할 수 없습니다"** 대체 페이지 표시
- Level 배지: `Level 4` (빨강)
- Camera 배지: `denied`
- 카메라 영상·큐브·자이로 기능 모두 비활성
- 안내 문구: "실제 서비스에서는 스탬프랠리 등 대체 참여 경로로 안내됩니다"

---

### 시나리오 4 — iOS 16.4 미만 단말 (가능 시)

**대상**: iOS 16.3 이하 (PWA `getUserMedia` 미지원 버전)

**단계**:
1. 홈 → "탐험" 탭 (또는 URL 직접 접속)
2. "테스트 시작" 탭

**기대 결과**:
- Camera 배지: `unsupported` 또는 `denied` (버전·브라우저 매체에 따라)
- Level 4 대체 페이지 자동 진입
- Errors 라인에 `mediaDevices.getUserMedia 미지원` 또는 관련 메시지

---

### 시나리오 5 — iOS PWA (홈화면 추가 상태)

**대상**: iOS 16.4+ 에서 "공유 > 홈 화면에 추가" 후 실행

**단계**:
1. Safari 에서 사이트 접속 → 공유 → 홈 화면에 추가
2. 홈 화면 아이콘으로 앱 실행
3. 하단 "탐험" 탭 → "테스트 시작"

**확인 포인트**:
- 권한 요청이 **Safari 브라우저 때와 동일하게** 떠야 함
- 자이로 권한이 PWA 컨텍스트에서 정상 동작하는지
- 카메라 스트림이 정상 표시되는지 (iOS 16.4 이전엔 PWA 내에서 차단됐음)
- 진단 패널 UA에 `Safari` 대신 iOS WebKit 문자열

**기대 결과**:
- 위 조건 전부 정상 작동해야 Phase 1 착수 가능
- 하나라도 실패 시 폴백 정책 재검토 필요 (보고 필수)

---

### 시나리오 6 — 저사양 단말 FPS 측정

**대상**: Galaxy A 시리즈 또는 저가 Android

**단계**:
1. 시나리오 1 과 동일 (정상 흐름)
2. 진단 패널의 FPS 값을 **10초간 관찰**, 평균 기록

**기대 결과**:
- FPS 30 이상: Level 1 유지 OK
- FPS 20~30: 경고 수준 — Phase 2 에서 저사양 폴백(`Level 3 2D 스프라이트`) 임계값 조정 근거
- FPS 20 미만: 현재 WebGL 렌더링이 부담. **보고 필수** — Phase 2 에서 자동 Level 3 다운그레이드 로직 추가해야 함

---

### 시나리오 7 — 백 버튼·재진입 테스트

**대상**: 임의 iOS/Android 1대

**단계**:
1. 시나리오 1 진행 → Level 1 카메라 뷰 도달
2. 좌상단 백 버튼 탭 → 이전 페이지 복귀 확인
3. 카메라 LED 꺼짐 확인 (스트림 정상 dispose)
4. 하단 "탐험" 다시 탭 → 재진입
5. 인트로 페이지부터 다시 시작되는지 확인 (상태 리셋)
6. "테스트 시작" 재탭 → 이번엔 권한 이미 허용돼 있어 다이얼로그 없이 바로 카메라 진입

**기대 결과**:
- 백 이동 시 리소스 정리 확실
- 재진입 시 이전 권한 유지 (iOS 16.4+ 에서는 세션 내 재요청 없음)
- 큐브 애니메이션·자이로 시차 모두 재진입 후에도 정상

## 2. 진단 패널 읽는 법

하단 반투명 패널에 표시되는 값:

| 키 | 의미 | 정상 범위 |
|---|---|---|
| Level | 진입한 폴백 레벨 | 시나리오별 기대값 |
| GPS / Camera / Orientation | 각 권한 상태 | `granted` / `denied` / `unsupported` / `not-required` / `prompt` |
| WebGL | 지원 버전 | `WebGL 2` 이상 권장 |
| GPU | 렌더러 이름 (가능 단말만 노출) | iOS는 숨김 많음 — null도 OK |
| Camera res | 후면 카메라 해상도 | 640×480 이상 |
| FPS | 실시간 프레임 | 30 이상, 이상적 55~60 |
| Safe Area | `T`(top) / `B`(bottom) 인셋 | iOS 노치 단말 T 값이 0 초과 |
| UA | userAgent 문자열 | 기기 식별용 |
| Errors | 에러 메시지 집합 | 비어 있어야 정상 |

**패널 토글**: "진단 패널 접기 ▾" 버튼으로 숨기고 다시 펼칠 수 있음.

## 3. 매트릭스 기록 가이드

`docs/ar-module/ar-test-matrix.md` 표에 단말별 1행 작성.

**예시 기록**:

```
| # | 단말 | OS / 브라우저 | GPS | Camera | Orientation | WebGL | FPS | Level | 이슈 |
| 1 | iPhone 14 Pro | iOS 17.3 Safari | granted | granted | granted | WebGL 2 | 60 | 1 | 없음 |
| 2 | iPhone 11 | iOS 17.1 Safari | granted | granted | denied | WebGL 2 | 58 | 2 | 자이로 거부 시 L2 정상 |
| 3 | iPhone 11 | iOS 17.1 PWA | granted | granted | granted | WebGL 2 | 57 | 1 | 홈화면 앱 정상 |
| 4 | Galaxy A54 | Android 14 Chrome | granted | granted | not-required | WebGL 2 | 42 | 1 | FPS 30~45 변동 |
| 5 | iPhone 8 | iOS 16.3 Safari | granted | unsupported | — | — | — | 4 | L4 폴백 정상 |
```

**기록 팁**:
- 같은 단말을 여러 조건(권한 허용/거부)으로 테스트할 땐 **행을 나눠서** 기록
- FPS는 10초 평균값
- 이슈가 있으면 `이슈` 컬럼에 간결히 (상세는 아래 이슈 보고 템플릿)

## 5. Phase 1 착수 판정 기준

다음 중 하나라도 실패 시 Phase 1 전 수정:
- [ ] iOS 16.4+ 단말에서 Level 1 진입 실패
- [ ] 자이로 권한 다이얼로그 자체가 표시되지 않음 (버튼 제스처 밖에서 호출되는 경우)
- [ ] 카메라 스트림이 표시되지 않음 (블랙 스크린)
- [ ] 백 버튼이 카메라 LED 를 끄지 못함 (리소스 리크)
- [ ] Android 중급 단말에서 FPS 20 미만

위 항목 전부 통과 + 매트릭스 최소 5행 작성 완료 시 Phase 1 프롬프트 진행.
